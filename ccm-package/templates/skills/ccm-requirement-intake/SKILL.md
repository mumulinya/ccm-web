# CCM Requirement Intake

## Workflow

1. Inventory every supplied source and distinguish readable content from unavailable or incomplete references.
2. Extract the business outcome, affected users, current behavior, desired behavior, constraints, dependencies, and explicit exclusions.
3. Preserve concrete contracts such as routes, methods, fields, states, permissions, errors, UI states, and examples with their source names.
4. Turn acceptance language into observable criteria without weakening it.
5. Separate facts from assumptions and missing decisions. Ask one focused question only when the missing answer blocks a safe plan.
6. Produce a compact intake containing source references, goal, scope, deliverables, constraints, acceptance criteria, assumptions to verify, and unresolved items.

Read [references/source-intake.md](references/source-intake.md) when the request contains multiple source types or remote document links.

Do not treat unreadable links, image filenames, or attachment presence as proof of their contents. Do not treat extracted requirements as execution authorization.
