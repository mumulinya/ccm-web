# 群聊主 Agent Plan Mode 4.0

目标：让群聊主 Agent 更接近 Claude Code / Codex / Cursor 的工作方式。复杂开发需求先进入只读探索和执行前计划，不直接改代码；只有低风险明确任务才自动继续，高风险、模糊、跨项目、删除/迁移/部署类任务必须等用户确认。

## 本轮实现

- 群聊项目任务创建前会生成 `cc-style-plan-mode` 执行前计划。
- 执行前计划会读取群聊上下文、项目代码快照、项目记忆/知识库召回和任务意图，形成：
  - 只读探索摘要
  - 影响项目和影响区域
  - 风险等级和确认原因
  - 验收标准
  - 权限边界
  - 子 Agent 自包含工作单要求
  - native session 续跑策略
- 高风险/模糊/跨项目/附件/删除迁移部署类任务：
  - 创建同一张任务卡和同一个 Trace
  - `intake_state = awaiting_confirmation`
  - 不入队、不派发子 Agent、不修改文件
  - 任务卡显示“确认执行 / 取消任务”
- 低风险明确任务：
  - 仍然生成执行前计划
  - 自动进入任务队列
  - 任务卡保留计划摘要，方便用户知道主 Agent 为什么继续
- 用户点击“确认执行”后：
  - 走 `/api/usability/intake/confirm`
  - 复用原 Task/Trace
  - 更新群聊同一张任务卡
  - 再进入执行队列

## 子 Agent 工作单增强

`buildDailyDevTaskDescription()` 已注入 Plan Mode 信息。底层 Claude / Codex / Cursor 项目 Agent 会收到：

- 执行前只读探索摘要
- 影响范围预判
- 风险与权限边界
- 子 Agent 工作单要求
- 会话续跑策略

这样子 Agent 拿到的是自包含工作单，而不是一句松散需求。

## 用户体验规则

- 普通询问不创建任务卡。
- 项目分析问题使用只读分析回复，不派发子 Agent。
- 明确开发任务才进入持久任务链。
- 任务卡只展示用户需要看的计划、风险、范围、验收和下一步。
- Trace、session、执行 ID 等内部细节仍折叠在技术详情。

## 自测覆盖

- 高风险计划必须等待确认。
- 低风险计划可以自动继续。
- 等待确认任务卡必须进入 `needs_user`。
- 等待确认任务卡必须提供 `confirm_plan` 操作。
- 任务卡必须渲染执行前计划。
- 前端确认按钮必须调用 `/api/usability/intake/confirm`。

## 后续可继续增强

- 用真实 E2E 场景验证跨项目计划确认、确认后派发、返工和最终验收。
- 把 Plan Mode 的只读探索来源做成可点击证据链。
- 对高风险任务支持用户编辑计划后再确认。
- 对自动继续的低风险任务增加“撤回/暂停”快捷入口。
