# 开发权限分级审批业务流程确认

Date: 2026-07-22

Status: Confirmed

## 基本原则

CCM 不给任何 Agent 永久的操作系统最高权限。项目子 Agent 默认拥有完成日常开发所需的项目内权限，额外权限采用精确申请、分级审批、限时授权、单次消费和完整审计。

## 审批顺序

```text
项目子 Agent 发现需要额外权限
  -> 确定性风险分类
  -> 项目内低风险：群聊主 Agent 策略批准
  -> 中风险：群聊主 Agent 模型判断必要性和影响
  -> 高风险或主 Agent 无法判断：来源会话与任务派发中心请求用户批准
  -> 宠物可用时播报提醒；飞书可用时向绑定会话发送提醒
  -> 生成绑定精确任务/项目/子 Agent 会话的限时租约
  -> 子 Agent 执行前消费租约
  -> 执行与审批写入任务时间线
```

## 默认项目权限

项目工作区内的读取、编辑、构建、测试、格式化、普通依赖安装和非破坏性 Git 检查属于日常开发范围。任务包仍限制允许路径、拒绝路径、网络、环境变量和验证命令。

## 群聊主 Agent 可审批

- 当前目标项目内、可恢复且完成当前任务确有必要的操作；
- 指定开发环境的网络访问；
- 普通 Git push 等具有远程影响但不覆盖历史的操作；
- 非生产部署或非破坏性数据库迁移，但必须有清晰范围和理由。

中风险审批调用当前群聊主 Agent 的统一模型。模型未配置、调用失败、返回不可靠或表示不确定时，不会本地猜测批准，而是升级用户。

## 必须由用户审批

- npm 或其他软件包发布；
- 生产环境部署；
- force push、历史覆盖和大规模破坏性删除；
- 密钥、Token、私钥等凭据读取或导出；
- 操作项目目录之外的文件；
- 系统安装、服务修改、管理员提权；
- 破坏性数据库迁移；
- 明显费用、影响范围不清或主 Agent 无法可靠判断的操作。

## 租约边界

群聊任务授权绑定 `taskId + groupId + gcs_* + project + tas_* + native session + operation + command/path/host checksum`；独立项目授权绑定 `project + project session + native session + operation + command/path/host checksum`。默认 15 分钟、一次使用，最多只能由审批方限制在 30 分钟和三次以内。兄弟任务、兄弟项目、兄弟会话和兄弟 Provider generation 不能复用。

项目子 Agent 通过独立内置 MCP `ccm__permission_broker.request_execution_permission` 申请。绑定命令的授权由 `execute_approved_command` 交给 CCM 受控运行器执行并消费；网络等非命令授权才调用 `consume_execution_permission`。`awaiting_user`、`rejected`、`expired` 和 `consumed` 均为禁止执行状态。

Codex 默认使用 `workspace-write` sandbox，包括 Windows；只有用户显式环境配置才能改成更高 sandbox。其他第三方 CLI 仍同时受其自身可用的 sandbox/approval 能力约束。

## 用户在哪里审批

- 全局 Agent 下发的任务：显示在发起任务的精确全局会话，同时显示在任务派发中心。
- 群聊主 Agent 下发的任务：显示在发起任务的精确 `gcs_*` 群聊会话，同时显示在任务派发中心。
- 独立项目 Agent：显示在当前精确项目会话，同时显示在任务派发中心。

宠物运行时可用时会播报“哪个项目请求什么权限”；飞书存在来源绑定时优先通知对应飞书会话，否则使用已配置的报告通道。两个通道分别记录送达状态，重试不会重复发送已经成功的通道。

如果任务来自飞书全局 Agent，会把申请 ID、项目、操作、风险与理由发送回原飞书会话。用户可以在该会话回复 `批准权限 perm_xxx` 或 `拒绝权限 perm_xxx`；后端必须同时匹配 `originType=global + 原全局会话 ID + awaiting_user` 才接受，兄弟飞书会话不能审批。批准后使用现有任务队列恢复原任务。

用户批准群聊或全局来源任务后，原任务自动回到执行队列；独立项目会话会在原会话内自动发送一条继续指令。新一轮 Agent 只能使用同一精确授权继续，不需要用户重新描述任务。
