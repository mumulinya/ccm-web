# 音乐意图与播放业务流程

Date: 2026-07-20

Status: Confirmed

## 模块关系

全局 Agent 和音乐 Agent 都可以发起播放，但不各自创建播放器。两者最终调用同一个常驻音乐播放引擎，同一时刻只允许最后一次播放请求生效。

音乐 Agent 没有会话列表。它使用单例对话记忆来理解音乐偏好，但播放动作仍经过统一播放规划器。

## 请求流程

1. Agent 判断用户是否明确要求播放；普通闲聊或只搜索歌曲不会自动播放。
2. 播放链保存完整用户原话，并同时携带 Agent 提取的点歌关键词。
3. 统一大模型把请求解析为结构化播放计划。
4. 播放器先查询本地曲库；没有合适候选时，根据当前模式调用 B 站或网易云搜索 API。
5. 候选通过对应选曲策略后进入唯一播放队列，较新的播放请求会覆盖尚未完成的旧请求。

## 五种播放策略

| 策略 | 示例 | 行为 |
| --- | --- | --- |
| `exact_song` | “播放周杰伦的晴天” | 严格校验歌名与歌手，不接受仅歌手相同的错误歌曲 |
| `artist_random` | “播放周杰伦的歌” | 先筛出歌手匹配项，再由模型结合完整原话选择；模型不可用时才随机兜底 |
| `mood_recommendation` | “我心情不好，放一首歌” | 模型把心情或场景转换为搜索主题，并从相关候选中推荐 |
| `genre_recommendation` | “来一首摇滚” | 按曲风、语言或音乐类型搜索和选择 |
| `random` | “随便放首歌” | 本地曲库优先随机；本地为空时使用外部 API 兜底 |

## 模型与兜底

统一大模型可用时，由模型理解歌手、歌曲、心情、活动和场景，不要求用户提供精确歌名。模型输出结构化计划，不直接伪造歌曲或音频地址。

模型暂时不可用时，后端使用确定性规则识别常见情绪、场景、曲风、歌手和随机播放请求。兜底只保证可用性，不能放宽精确点歌的匹配门禁。

歌手请求会先用候选的歌手、标题等可验证字段排除其他歌手，再把剩余作品交给模型。这样模型负责理解和选曲，但不能越过歌手范围选择无关歌曲。

## 数据边界

- 完整用户原话只用于本次意图规划和选曲，不创建音乐会话。
- 音乐偏好可由音乐 Agent 的长期记忆提供，但不会把全局 Agent、群聊或项目 transcript 混入音乐上下文。
- 远程播放队列保存 `request_text`，保证 Web client effect 与后台轮询使用同一请求语义。
- 外部搜索结果必须来自已有音乐 API；模型只能选择候选，不能生成不可验证的播放地址。
