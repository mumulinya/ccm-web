# CCM Agent 推理—执行—验证闭环 6.0

> 本文是全局主 Agent 与群聊主 Agent 的持续开发记录。实现和验收证据应随代码同步更新。

## 目标

让大模型负责理解、规划和动态判断，同时由确定性系统负责授权、目标落地、状态机、幂等执行、事实复核、验证门禁和审计。系统不能退化为关键词派发，也不能让模型仅凭记忆或子 Agent 回执自行宣告完成。

## 设计原则

1. 原始目标永远保留；澄清只形成新的有效目标，不覆盖原始请求。
2. 同一待澄清运行在短时间内续跑同一个 run ID；明确开始新话题时不继承旧运行。
3. 授权只能在同一澄清链的明确范围内继承；“只分析/不执行”等当前限制立即撤销写入授权。
4. 行动前读取并记录当前真实状态，行动后把工具结果变成验证断言和证据。
5. 事实变化、工具失败、依赖输出不一致或验收缺口必须产生偏差记录并触发重规划。
6. 服务重启、原生会话恢复和执行器切换不是机械续跑；必须重新核对原始目标、当前代码状态、剩余缺口与验收条件。
7. 计划、事实、断言、偏差、恢复复核和决策理由都进入可审计账本与页面展示。
8. 大模型不可用时，规则回退不得解释普通自然语言后执行写操作；只允许确定性只读检查，或执行已经结构化且来自可信任务/定时工作流的工作单。

## 统一推理账本

`backend/agent-reasoning-loop.ts` 提供共享数据模型：

- `original_goal`：不可变的原始目标。
- `effective_goal`：原始目标加当前澄清链形成的有效目标。
- `authorization_scope`：本轮明确允许的目标与影响范围。
- `clarification_chain`：问题、用户补充和时间。
- `plan_version / plan_history`：计划版本、变化原因和依赖的事实哈希。
- `replan_required / last_replan_reason`：事实变化或失败后由服务端强制置位，只有新计划或成功的纠正行动才能消解。
- `fact_snapshots`：行动前/后真实状态及来源哈希。
- `assertions`：目标、文件、验证和最终验收是否已被证据证明。
- `deviations`：工具失败、事实不一致、验收缺口和提前完成等偏差。
- `postmortems`：误判触发点、发生了什么、当前纠正动作和防止重复的约束。
- `recovery_checks`：恢复时是否重新核对目标、状态、验收与剩余缺口。
- `explanations`：每次回答、询问、派发、确认和完成的理由。

## 已实现

### 全局主 Agent

- 每个 Agentic run 自动创建推理账本，并把当前系统上下文和工具结果记录为事实快照。
- 模型每次生成的新计划都会比较旧计划；发生变化时增加 `plan_version` 并记录原因和事实哈希。
- 低置信度或目标不明时进入澄清；同一 session 在 30 分钟内的下一条补充会续跑原 run ID。
- 澄清链保留原始目标；当前回复若包含“不执行/只分析”等限制，不继承写权限。
- 明确执行意图却没有形成工具行动时，系统阻止把说明文字当完成并继续询问。
- 工具异常、返回 `success=false` 或模型在失败后试图结束时记录偏差，禁止无证据完成并要求重规划。
- 漏执行、提前完成、工具失败和验收缺口会自动形成结构化复盘，记录纠正动作与防重复策略，而不是只留一条错误日志。
- 服务重启恢复运行前重新获取当前系统上下文，并记录恢复复核。
- API 返回完整 `reasoning_loop`；全局助手运行卡片显示计划版本、事实、待证明断言、偏差、澄清轮次和最近决策理由。

### 群聊主 Agent

- LLM 协调协议新增 `reasoning`：已知事实、待核验假设、验证断言、依赖理由和重规划触发条件。
- 普通群聊在 LLM 配置缺失或调用失败时不再由关键词规则自动派发；coded fallback 只允许承接含任务标题、业务目标和验收标准的可信结构化任务/定时工作单。
- 跨项目依赖必须基于接口契约、数据流、冲突范围或验收顺序，而不是前后端关键词机械排序。
- 任务执行前生成 preflight 快照，核对业务目标、验收标准、执行状态和任务会话。
- 主 Agent 的协调阶段写入计划版本、语义事实和待证明断言。
- 服务重启恢复、原生 session 重试和 Claude/Codex/Cursor 执行器切换前，重新核对并向 Worker 注入原始目标、当前状态、验收条件和剩余缺口。
- 交付摘要持续更新目标、文件、独立 Runner 验证和最终验收断言；验收缺口进入偏差列表。
- 群聊内联状态显示计划版本、待证明断言和偏差；协作看板展示完整计划、断言、偏差与恢复复核。

## 自动化覆盖

- 多轮澄清续跑同一 run ID。
- 原始目标不被澄清覆盖。
- 澄清后的计划、事实和工具验证可审计。
- 计划变化递增版本。
- 授权范围显式记录。
- 偏差记录和恢复目标/验收复核。
- 群聊语义事实、假设、验证断言、依赖理由和重规划触发条件归一化。
- 既有原生会话、外部 Runner、冲突 worktree、最终门禁和可靠性回归继续通过。

- 原始目标仍为“帮我优化一下”，有效目标追加了澄清；计划从 v1 更新为 v2，并记录当前系统上下文和 `inspect_project` 的事实快照。

## 已知限制与后续方向

- 当前事实快照以紧凑 JSON 和哈希存储，尚未对同一对象生成字段级 diff；下一步可增加事实变化解释。
- 30 分钟澄清窗口是确定性安全边界；未来可用轻量语义模型判断“这是澄清还是新话题”，但仍要保留显式新话题逃生口。
- 当前已支持以“新问题 / 换个问题 / 忽略刚才 / 重新开始”等显式前缀绕过旧澄清链；自然语言的新话题判断仍优先保持保守，避免误继承授权。
- 动态重规划目前由事实/失败提示驱动并记录版本；后续应建立离线 Trace 回放，评估重规划是否真的比旧计划更好。
- 群聊旧任务没有推理账本，只有在再次执行、恢复或重新生成交付摘要后才会补齐，质量指标应按 6.0 数据窗口解释。
