# 日常使用体验 11.0

## 目标

让普通用户只需要表达目标、确认计划和处理真正的阻塞，不必理解 `CCM_AGENT_RECEIPT`、scratchpad、Trace、执行内核或内部调度协议。

## 本轮实现

### 1. “我的工作台”成为默认入口

- 默认首页由“项目管理”改为“我的工作台”，并移动到核心功能首位。
- 聚合群聊、项目、普通任务、派发任务、定时任务和交付报告。
- 首页只突出：需要用户决定、执行失败、正在推进、等待开始、刚刚完成。
- 优先处理区最多展示 5 项，避免历史阻塞淹没当前工作。
- 技术协议和系统细节默认折叠。

### 2. 统一工作台后端

- 新增 `GET /api/usability/workbench`，由服务端统一排序和解释任务状态。
- 状态转换为用户语言：需要你决定、执行遇到问题、正在推进、等待开始、刚刚完成。
- 通知只由三类事件产生：需要决定、失败、完成。
- 已完成或已取消超过 30 天的任务由后台每 6 小时自动归档。
- 已取消任务不会再因旧失败摘要被误判为待处理。

### 3. 执行前确认卡

- 新增 `POST /api/usability/intake/preview`：创建不可执行的需求草稿，补全目标项目、影响范围、验收标准和风险。
- 新增 `POST /api/usability/intake/confirm`：用户确认后复用原 Task ID 和 Trace ID 入队。
- 确认前明确提示“尚未开始，不会修改项目”。
- 放弃确认卡会归档草稿，不会留下待执行任务。

### 4. 上下文操作和直接定位

- 工作台任务卡支持确认、补充、暂停、恢复、重试、取消、归档和查看交付。
- 从工作台打开任务时，任务管理页直接打开对应任务的交付报告，而不是只跳到任务列表。
- 失败卡提供重试和切换执行器入口；需要补充的任务提供补充说明和恢复入口。

### 5. 首屏性能和信息降噪

- 原应用会通过 `v-show` 一次挂载所有页面，导致首页背后同时运行群聊、全局 Agent、任务页等轮询。
- 现改为标签首次打开时才挂载，之后保留页面状态。
- 首次打开工作台不再加载隐藏页面的长对话、监控请求和定时刷新。

## 验收证据

- TypeScript：`npm run check` 通过。
- 前端生产构建：`npm run build:frontend` 通过。
- 后端生产构建：`npm run build:backend` 通过。
- 工作台聚合实测：6 个项目、3 个协作群、4 个定时任务；优先区限制为 5 项。
- Task/Trace 连续性实测：测试任务 `mr1fo6xaxphg` 从预览、持久化到确认入队始终使用 Trace `task_mr1fo6x9_c26edfe0a77e`，`same_task_trace=true`。
- 测试任务确认后立即取消，最终状态为 `cancelled`，未产生项目文件修改。
- 生命周期任务 `mr1g6770o0lq` 实测完成草稿、补充说明、暂停、恢复入队、受控取消和归档；补充记录为 1 条。
- 验收发现“待决定超过 5 条会挤掉失败事项”，已修正为失败优先、待决定其次。
- 失败优先级复验任务 `mr1g8oxfs0s0` 在工作台返回 `phase=failed`，并提供 `retry,switch_executor,cancel` 三个恢复动作；复验后已归档清理。
- 浏览器验收确认默认首页、自然语言输入、资源选择、优先处理、正在推进、刚刚完成和折叠技术详情均可见。
- 浏览器验收发现并修复：确认卡字段首次映射为空、取消任务误入待处理、首屏挂载所有重组件三项问题。

## 当前边界与下一步

- 当前需求补全采用可靠的规则提炼；下一轮可以接主 Agent 推理接口，在规则结果基础上进一步补全范围和验收标准。
- 工作台已提供关键动作，但任务页和群聊页仍保留较多技术视图；后续可继续统一为同一张用户任务卡。
- 生产包仍是单个较大的前端 chunk，后续可按标签页做动态 import，进一步降低首次下载和解析成本。
- 下一轮应基于真实开发群完成一次不取消的完整任务，从确认卡一直走到交付卡，并记录用户补充、暂停/恢复、失败恢复和最终验收的时间线。
