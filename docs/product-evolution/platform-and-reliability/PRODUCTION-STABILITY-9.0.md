# 生产级无人工重启稳定性验收 9.0

## 目标

本轮将旧版“服务能在多次重启中恢复”的浸泡测试升级为“冻结代码和配置、单一 CCM 进程连续运行 24 小时”的生产验收。

## 新增能力

### 1. 进程生命周期账本

- 记录每次启动、退出、退出码、信号、PID、Boot ID 和前一实例。
- 重启分类：`manual_restart`、`code_reload`、`system_shutdown`、`unexpected_crash`。
- 支持在受控重启前写入 restart intent，下一实例不会把人工重启误报为崩溃。
- 未留下正常退出记录的死进程会在下一次启动时登记为 `unexpected_crash`。
- 生命周期记录位于 `reliability/process-lifecycle/`。

### 2. 稳定性债务清理

- 失效幂等操作被结算为可审计的 `failed/reconciled`，保留 Trace。
- 失效任务租约被安全释放为 `orphaned_reconciled`。
- 超过 30 分钟、处于 `in_progress` 且没有有效租约的任务恢复为 `pending`，不删除任务。
- PID 已失效的飞书锁文件会被清理，存活进程的锁不会动。

### 3. 更准确的监控

- 启动宽限期内的事件循环延迟记为 `info`，不混入运行期告警。
- 非启动期超过 500 ms 为 warning，超过 2000 ms 为 critical。
- 同一实体的连续异常形成一个 incident；报告同时展示独立事故数和采样命中数。
- 飞书健康要求全局控制机器人以及所有配置为 `feishu` 的预期项目机器人全部在线。
- 每个异常携带 Task、Trace、PID、Boot ID 或实体 ID。

### 4. 干净 24 小时模式

- 启动前清理历史债务并执行 preflight。
- 冻结 CCM 构建产物、项目/群聊配置和机器人 TOML 配置；运行期间哈希变化直接失败。
- 记录计划内与非预期重启，干净模式不允许任何 CCM Boot ID 变化。
- 内存报告使用同一进程预热后的线性回归斜率，不再用跨重启的首尾 RSS 冒充泄漏趋势。
- 报告包含非启动期事件循环 P95、P99、峰值和最长单进程连续运行时间。

## 一键命令

```powershell
cd C:\Users\admin\.cc-connect\ccm
npm run stability:9:start
npm run stability:9:check
```

`stability:9:start` 会启动尚未运行的预期飞书项目机器人、清理稳定性债务并启动干净 24 小时验收。若前置条件不满足，它会退出失败并打印具体缺口。

## 完成门禁

- 24 小时内 CCM Boot ID 唯一，非预期重启为 0。
- 无新增失效任务租约、失效幂等操作和无租约长时间任务。
- Agent Runner 与所有预期飞书机器人采样可用率 100%。
- 代码和配置冻结哈希不变。
- 非启动期无 critical 事件循环延迟。
- 最终报告由同一个 Boot ID 连续覆盖至少 23.9 小时。
- 进程预热后 RSS 线性增长不超过 8 MiB/小时（默认采样下至少观察 60 个预热样本）。
- 故障演练保持通过，采样错误为 0。

测试启动后，在本文件追加 Test ID、开始/结束时间和最终报告路径；在真实 24 小时证据产生前不得把 9.0 标记完成。
