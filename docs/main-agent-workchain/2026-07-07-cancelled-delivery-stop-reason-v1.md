# 取消态交付报告停止原因 v1

## 背景

参考 `D:\claude-code` 的协调/Agent 通知链路，子 Agent 或工作线程的结果会用 `completed / failed / killed` 这类 outcome 表达，`summary` 里会区分完成、失败和停止。停止不是失败风险，也不应该在用户主视图里被包装成“风险与待确认”。

本项目上一版已经让取消态的“接下来建议”主动作变成“重新发起需求”，但统一交付报告里仍可能出现“风险与待确认：用户取消了本轮任务”。这会让用户误以为取消是一种需要修复的风险。

## 本次升级

- 取消态交付报告的原因区从“风险与待确认”改成“停止原因”。
- 取消态完成总览的指标标签从“风险”改成“停止原因”。
- Pickup 摘要中取消态的原因前缀从“注意”改成“停止原因”。
- 前端兜底生成的全局取消态交付报告也使用“停止原因”。
- 任务卡在没有后端 completion card 时，本地总览同样使用“停止原因”。

## 用户可见规则

- 完成态：展示完成内容、涉及范围、验证结果、风险与待确认、接下来建议。
- 失败态：展示处理结果、涉及范围、验证结果、未完成原因、接下来建议。
- 取消态：展示停止说明、涉及范围、验证结果、停止原因、接下来建议。
- 取消态的主建议仍是重新发起或恢复需求，不把停止原因当成返工入口。

## 验收点

- 取消态报告不再出现“风险与待确认”。
- 取消态报告出现“停止原因”。
- 取消态 completion card 的指标标签为“停止原因”。
- 普通问话仍不展示任务卡或 Todo。
- 失败态仍保留“未完成原因”。
