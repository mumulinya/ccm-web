# 定时任务生产可靠性闭环 v2

日期：2026-07-13

## 目标

将定时任务从“按时创建任务”升级为可长期无人值守的运行闭环，覆盖显式时区、停机补跑、失败自动重试、启动恢复、运行级控制、飞书通知、批量管理和真实浏览器验收。

## 可靠调度

- 每个任务持久化 IANA 时区，默认 `Asia/Shanghai`；Cron 匹配和下一次运行都按任务时区计算，不依赖服务器本地时区。
- 调度由持久化 `next_run` 驱动，不再只检查进程当前分钟是否匹配表达式。
- 服务停机错过时间后支持两种策略：恢复后补跑一次，或记录一条跳过运行。
- 补跑策略可设置最长窗口，超过窗口不会补建过期任务。
- 计划触发继续使用持久化幂等记录，同一任务同一计划分钟只会认领一次。

## 失败重试

- 普通项目任务、普通群聊任务和业务开发任务统一支持 `0-10` 次额外重试。
- 每个任务可配置 `1-1440` 分钟重试间隔。
- 关联任务失败后，运行进入“等待重试”，保存 `next_retry_at`。
- 到期后创建带 `parent_run_id` 的新运行尝试，并复用原任务及其历史证据，不创建不可追踪的平行任务。
- 重试前先把任务绑定到新运行，再加入执行队列，避免快速失败回写到旧运行。
- 禁用或归档定时任务会同时停止自动重试；用户仍可从历史记录手动重试。

## 启动恢复

- 服务启动时扫描未结束运行。
- 有关联任务的运行根据任务真实状态重新聚合，不以旧页面状态为准。
- 超过 5 分钟、仍停留在触发中且没有成功创建任务的运行会标记为启动恢复失败，并按任务重试策略进入恢复流程。
- 缺少 `next_run` 的已启用旧任务会自动补齐下一次计划时间。

## 运行级控制

新增接口：

- `POST /api/cron/run/retry`：重试失败运行。
- `POST /api/cron/run/resume`：继续等待或已取消运行。
- `POST /api/cron/run/cancel`：取消本轮及其未完成任务。

取消操作会移出队列、写入执行取消请求、取消 TestAgent 运行并持久化任务和运行状态，不是只修改界面标签。重试和继续会生成新的尝试记录，保留父运行关系。

## 飞书通知

- 每个定时任务可独立启用通知。
- 可选择开始、完成、失败、等待处理、停机补跑和取消事件。
- 复用设置页面配置的飞书通知机器人。
- 每个事件在运行记录中保存 `sending/sent/failed` 回执和错误摘要，重复刷新不会重复发送同一事件。

## 运行记录

每轮新增或规范化字段：

- `scheduled_for`
- `attempt`
- `parent_run_id`
- `next_retry_at`
- `retry_child_run_id`
- `recovered_after_restart`
- `notifications`

旧版本只保存最后状态、没有逐轮事件的数据会标记为“历史摘要”。页面明确说明它不能作为完整运行回放，不伪造不存在的过程。

## 页面

- 新增名称/目标/提示词搜索、状态筛选和目标类型筛选。
- 接通后端批量启用、禁用、归档和恢复。
- 新建与编辑表单可配置时区、重试次数、重试间隔、错过任务策略、补跑窗口和通知事件。
- 运行记录抽屉提供重试本轮、立即重试、继续运行和取消本轮。
- 技术字段和通知回执默认收在技术详情。
- 手机任务列表改为纵向信息布局，不需要横向拖动表格；运行控制和 TestAgent 证据入口保持可用。

## 验证

- `npm run check`：通过。
- `npm run build`：前端、飞书 MCP 和后端完整构建通过。
- `npm run test:cron-reliability`：通过时区匹配、下一次运行、40 条保留、历史摘要、停机跳过、到期自动重试、手动重试、启动恢复、取消任务和配置校验。
- `npm run test:work-journal`：日报周报工作证据归属和去重通过。
- `npm run test:cron-run-history-render`：不再使用静态夹具；在隔离用户目录启动真实后端和真实应用，验证搜索、批量禁用/启用、运行记录、可靠性表单、技术详情默认折叠和桌面/手机无页面级溢出。

## 数据边界

生产任务文件、生产任务状态和飞书配置没有用于回归写入。API 与浏览器测试都在临时用户目录运行，并在结束后删除隔离数据。旧版本未保存的历史轮次无法从现有文件推导，因此只保留诚实的兼容摘要。
