# 工作项派发反馈 v1

## 目标

参考 Claude Code 的任务认领与依赖处理方式，让群聊主 Agent 和全局主 Agent 在派发子 Agent 工作项时明确告诉用户真实结果，而不是只显示笼统的“已提交”。

## 本次实现

- 新增统一的 `ccm-main-agent-work-item-claim-summary-v1` 用户摘要。
- 覆盖已派发、依赖未完成、Agent 正忙、已被认领、已经结束、工作项不存在六类结果。
- 用户区只展示“为什么暂未派发、接下来会怎样”等自然语言。
- `agent_busy`、`blocked`、工作项 ID、依赖引用等原始字段只保留在技术详情。
- 群聊“继续派发”现在会传递准确的 `work_item_id`，并根据后端真实结果展示成功或等待提示。
- 后端在续跑前执行无副作用预检。依赖未完成或 Agent 忙时不创建错误续跑，返回 `waiting: true`、`queued: false`。
- 预检通过后才续跑并正式认领工作项，认领结果持久化到 `work_item_runtime.last_claim_summary`。
- 群聊任务卡和全局任务卡复用同一份派发状态组件。

## 用户可见行为

- 认领成功：显示“某 Agent 已接下某工作项”，并说明会继续跟踪执行和验证。
- 依赖阻塞：显示正在等待哪个前置工作，前置完成后自动刷新队列。
- Agent 忙碌：显示当前正在处理什么，当前工作项继续等待。
- 重复派发：说明工作项已经在处理中，不会重复派发。
- 已结束：说明不需要再次派发。
- 队列变化：说明未找到对应工作项并刷新可派发列表。

普通问话仍不会展示工作项或 Todo。

## 回归覆盖

- 后端工作项自测覆盖六类认领结果和技术字段隔离。
- 静态 UI 自测覆盖组件、数据映射、群聊操作参数和后端持久化。
- Playwright 真实渲染同时覆盖群聊与全局任务卡：
  - 展示友好的“派发状态”和下一步。
  - 不展示 `agent_busy`。
  - 不展示工作项内部 ID。
