# 主 Agent 失败文案责任归属 v1

## 背景

全局主 Agent、群聊主 Agent 和共享任务卡在遇到内部错误、复核失败或证据缺口时，部分兜底文案会显示“需要处理”“需处理”或“待处理缺口”。这些状态通常应由主 Agent 继续排查、安排返工或补齐证据，不应让用户误以为自己必须立即操作。

## 本次升级

- 内部执行错误统一显示为“待排查”，并明确由主 Agent 继续定位；协议、Trace、会话和原始错误仍放在默认折叠的技术详情中。
- TestAgent 复核失败统一显示“未通过项”“待返工”或“待补齐项”，不再使用归属不清的“待处理缺口”。
- 交付报告中的失败验收项显示“未通过”，计划或证据缺口显示“待补齐”。
- 无风险状态改为“暂无需要额外关注的风险”，避免暗示存在用户待办。
- 群聊与全局共享任务卡同时增加真实渲染断言，确认内部错误不会暴露协议字段，也不会显示成“需要处理”。

## 用户可见效果

- 系统自己要继续排查：显示“待排查”以及主 Agent 会做什么。
- 执行成员或 TestAgent 需要返工：显示“未通过项 / 待返工”。
- 只有确实需要用户确认、授权或补充资料时，才显示“等你确认”或“待你补充”。

## 验证范围

- 主 Agent 决策 UI 源码自测。
- 共享任务卡数据自测。
- 后端类型检查与前端、后端构建。
- Playwright 渲染回归：同一内部失败数据分别以全局和群聊上下文渲染，技术详情默认折叠。
