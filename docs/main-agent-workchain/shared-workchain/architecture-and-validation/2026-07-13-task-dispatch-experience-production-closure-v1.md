# 任务派发体验生产闭环 v1

日期：2026-07-13

## 目标

整理任务派发页面的信息层级，确保普通用户先看到任务状态和下一步操作，队列恢复、CLI 进程和运行债务等技术治理能力只在明确进入“运行管理”后展示。同时保持创建、派发、续跑、验收、回放和 Test Agent 复核链路可用。

## 页面结构

任务派发页面现在分为三个稳定视图：

1. **任务概览**：展示正在推进、已完成和需要用户处理的最近执行记录。
2. **全部任务**：提供搜索、状态筛选、归档切换和批量操作。
3. **运行管理**：集中放置队列恢复、失败恢复、批量入队、执行通道复检、Agent CLI 进程和运行债务治理。

顶部只保留一个“新建任务”入口，菜单内区分：

- 业务开发任务：交给群聊主 Agent 拆分、派发并组织验收。
- 普通任务：快速分配给指定群聊或项目。

“删除”语义已经收敛：可恢复操作统一显示为“归档”，只有归档页面中的不可恢复操作显示“永久清除”。

## 用户可见内容

- “主 Agent 计划”改为“执行计划”。
- “子 Agent 执行”改为“协作进展”。
- “阻塞/接管”改为“需要关注”。
- CLI 探针状态、项目失败明细、修复步骤和执行内核状态默认折叠在“技术详情”。
- 普通问话不显示决策卡、Todo 或派发摘要，只显示自然回复。
- 任务型请求仍展示计划、实时状态、验收提醒、Test Agent 复核和最终总结。

## 响应式修复

原手机端任务容器可视宽度为 367px，但内部内容宽度达到 799px，顶部操作有一半不可达。本次改造后：

- 任务容器和文档宽度均有自动化横向溢出断言。
- 统计、创建操作、视图切换、筛选器和运行管理操作在 390x844 视口内重新排布。
- 概览任务详情改为单列，全部任务保留完整操作且不需要横向拖动。

## 功能与接口审计

任务派发前端依赖的 36 条路由契约全部能在后端找到处理器，覆盖：

- 任务创建、更新、归档、恢复、永久清除和批量操作。
- 任务入队、批量入队、队列恢复、看门狗和执行失败恢复。
- 业务开发需求池导入、状态更新、单条派发和批量派发。
- 执行看板、继续执行、按缺口续跑、取消和重派。
- 执行隔离现场的回滚、合并和清理。
- Agent 运行进程、执行通道探针、运行债务和任务回放。

当前运行服务的 7 条关键只读接口已真实请求通过：任务 31 条、最近执行记录 12 条、需求池 3 条，队列、看门狗、Agent 运行和任务回放均返回有效结构。

## 回归修复

原主 Agent 决策 UI 自测的 9 项失败已全部解决：

- 8 项来自渲染回归已经增加到 38 张截图，但聚合自测仍检查旧的 37 张基线。
- 1 项来自记忆中心残留“回执”用户可见术语，已改为“结果说明”。

主 Agent 38 张真实渲染回归还修正了一个旧断言：普通问话应隐藏决策卡，而不是仅隐藏卡片中的 Todo。
