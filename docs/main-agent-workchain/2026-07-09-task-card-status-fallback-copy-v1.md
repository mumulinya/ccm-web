# 共享任务卡状态 fallback 文案

日期：2026-07-09

## 背景

群聊主 Agent 和全局主 Agent 都会复用共享任务卡。部分结构化数据没有显式 `status_label` 或 `activeForm` 时，前端 fallback 会把 `blocked`、`failed`、恢复缺口渲染成“需处理 / 需要处理 / 仍需处理”。这容易让用户误以为这些都是自己的待办。

## 本次调整

- 执行成员进展里的 `blocked` fallback 从“需处理”改为“待补齐”。
- 子执行成员状态构建里的 `blocked` fallback 从“需处理”改为“待补齐”。
- 工作项 active form fallback 拆分：
  - `blocked`：待补齐
  - `failed`：待排查
- 恢复接续里的剩余缺口从“仍需处理”改为“仍待补齐”。
- 跨项目任务卡构建时，阻塞执行目标默认描述为“待补齐”，不再写成“需要处理”。
- 新增共享任务卡状态 fallback 渲染 fixture 和 Playwright 断言，防止这些文案回退。

## 验收点

- 群聊和全局任务卡缺少显式状态文案时，也能展示友好的“待补齐/待排查”。
- 真正等待用户确认的卡片仍保留“需要你处理/需要你确认”。
- 内部缺口不会冒充用户待办。
