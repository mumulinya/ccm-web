# 交付验收证据口径 v1

## 背景

群聊主 Agent 和全局主 Agent 已经会整理最终交付报告、完成总览和任务卡，但“验证结果”过去主要展示命令或检查列表，用户不一定能分辨哪些是实际执行、哪些只是建议、哪些是失败或缺失的验证证据。

对照 Claude Code 的交付链路，验证结果不能被包装成“完成证明”；它只是验收证据的一部分。没有实际执行、外部 Runner 证据缺失、项目必需验证缺口或失败验证，都应该以用户能看懂的方式展示出来，原始日志和协议细节仍放在技术详情里。

## 本次升级

- 新增 `ccm-main-agent-verification-evidence-v1`，在交付报告中单独生成“验收证据”。
- 区分展示：
  - 已实际执行的验证。
  - 失败验证。
  - 仅建议或未执行的验证。
  - 项目必需验证命令缺口。
  - 外部 Runner 证据数量和来源门禁。
- 完成总览里的“验证”指标改为优先展示证据口径，例如“1 项实际执行”“缺 1 项”“仅建议”。
- 任务卡交付报告区块上限从 9 个扩到 10 个，确保新增“验收证据”不会挤掉“下一步”。
- 截图回归 fixture 和 Playwright 断言新增“验收证据”可见性检查，覆盖群聊任务卡和全局历史任务卡。

## 用户可见策略

- 主文本只展示友好结论，例如“已实际执行 1 项验证”“外部 Runner 证据 1 项”。
- 不展示 `trace_id`、原始 payload、结果协议、内部工作单等技术字段。
- 技术排障信息继续默认折叠在“技术详情”里。
- 普通问答仍不会显示 Todo、交付报告或验收证据卡。
