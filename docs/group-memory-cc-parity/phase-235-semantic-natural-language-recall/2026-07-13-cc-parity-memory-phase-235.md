# CCM Memory CC Parity Phase 235

## 阶段目标

提升群聊会话类型化记忆对低唯一性自然语言约束的召回能力，并继续收紧会话边界：

- 不依赖哨兵字符串、文件路径或完全相同措辞，也能召回语义一致的用户约束。
- 正确区分“禁止”“必须”“允许”“先后顺序”和条件语义。
- 反向规则必须降权，不能因为共享“接口、失败、重试”等词而排在正确规则之前。
- 同一次召回中的同义约束只注入一次，减少子 Agent 上下文浪费。
- 每个 `groupId::gcs_*` 仍是独立记忆范围。
- 项目子 Agent 只接收所属群聊会话的记忆。
- 全局 Agent 只接收全局记忆和路由目录，不接收群聊会话正文。
- 旧群聊会话直接删除，不迁移。

## Claude Code 对照结论

Claude Code 的 `MEMORY.md` 是稳定入口，实际内容按当前任务相关性加载。CCM 原有召回主要依赖原始 token/子串匹配、固定类型权重、路径条件和专用压力策略；这对路径、命令和唯一标识很有效，但普通中文规则容易被泛化项目记录挤掉。

Phase 235 在现有确定性策略上增加可审计的 BM25-lite 语义参考层。语义层只能补充相关性，不能覆盖路径条件、强制召回、上下文压力降权或 ignore-memory。

## 实现内容

### 1. 自然语言语义特征

在 `group-memory-index.ts` 增加中英文概念归一化：

- 验证、代码修改、失败、接口、重试、人工确认。
- 记忆、上下文、会话、压缩、恢复。
- 文件、测试、部署、安全、持久化、路由。
- 子 Agent、全局 Agent 和用户指令。

约束极性包括：

- `prohibit`
- `require`
- `allow`
- `sequence`
- `conditional`

关系签名包括：

- `verify_before_code_change`
- `approval_before_retry`
- `retry_allowed`

修复了中文“不允许”同时误命中“允许”的问题。英文 `can/may` 只有紧邻明确动作词时才视为授权，避免把模板中的 `can recover` 当成允许规则。

### 2. 可解释语义评分

每个候选文档产生 `ccm-group-typed-memory-semantic-reference-score-v1`：

- 文档频率加权的词汇重合。
- 归一化概念重合。
- 同一句内概念共现。
- 约束形状和极性匹配。
- 禁止与允许的冲突惩罚。
- 查询/文档概念、极性、关系和逐项原因。

正向语义增益上限为 18，反向惩罚下限为 -20。这样自然语言约束可以补足原始召回，但不会压过硬边界。

当原有专用策略已把候选降到非正分时，普通正向语义增益默认被门禁；只有当前查询和文档都明确是自然语言约束、至少匹配两个概念时才能恢复。压力记忆、post-compact 使用反馈等专用降权因此仍然有效。

### 3. 同义约束去重

同批召回按以下证据抑制重复：

- 相同文件。
- 同类型文档的约束关系签名一致。
- 同类型文档的核心概念高度重合且授权方向一致。
- 同类型文档 token Jaccard 高相似。

路径条件已命中的文档和强制召回文档不参与同义抑制。不同记忆类型默认不互相去重，避免把 failure memory 和 reference evidence 因共享蒸馏模板而错误合并。

### 4. 硬路径优先级

命中 `paths` 条件的记忆权重提升到 64，并免于同义抑制。`src/pay.ts` 专属规则会排在通用项目记忆之前；未命中的条件文档仍然 fail closed。

### 5. 会话边界修复

全局记忆仲裁蒸馏触发二次类型化召回时，原代码误用了裸 `groupId`。现已改为当前 `typedMemoryScopeId`，即：

```text
groupId--gcs_*
```

因此二次召回也不能退回群聊级共享目录，不会跨会话读取旧类型化记忆。

### 6. Memory Center 可观测性

Memory Center 新增：

- 语义匹配数量。
- 极性冲突降权数量。
- 同义去重数量。
- 匹配文档、调整值、概念和 duplicate target。

界面只展示诊断元数据，不展开记忆正文。移动端诊断行使用单列布局。

## 关键回归样例

查询：

```text
接口出错时不要再次尝试，先让人确认
```

正确记忆：

```text
接口调用失败后禁止自动重试，必须等待人工确认。
```

反向记忆：

```text
测试失败后可以直接自动重试接口三次，然后再通知人工。
```

结果：

- 正确规则得分 31。
- 反向规则得分约 8.743，并记录 `polarity_conflict_prohibit_vs_allow`。
- 同义接口规则只注入一条。
- 两条“修改代码前先验证”的同义规则只注入一条。
