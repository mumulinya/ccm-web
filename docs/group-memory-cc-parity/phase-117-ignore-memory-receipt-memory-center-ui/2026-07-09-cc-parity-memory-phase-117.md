# Phase 117 - Ignore-Memory Receipt Memory Center UI

## Goal

Continue the long-running CCM memory upgrade toward Claude Code style context discipline by making `ignore memory` compliance visible in Memory Center.

Phase 115 added the WorkerContextPacket ignore-memory policy, and Phase 116 audited whether child Agents complied through `CCM_AGENT_RECEIPT.memoryIgnored`. This phase adds a first-class Memory Center surface so the group main Agent can see, refresh, and triage those checks across multiple group chats.

## Changed

- Added a dedicated Memory Center panel for `worker_context_packet_ignore_memory_receipt_compliance`.
- Added the paired repair-work-item check `worker_context_packet_ignore_memory_receipt_repair_work_items` to the same panel.
- The panel summarizes:
  - coverage rate across checked groups,
  - ignore-policy binding count,
  - compliant receipt count,
  - missing child receipt / missing `memoryIgnored` gaps,
  - `memoryUsed` violations under ignore policy,
  - open, current, stale, and total repair work items.
- Added targeted refresh buttons for both checks.
- Added row-level group summaries for compliance and repair state.
- Added responsive and dark-mode styles for the new panel.

## Long-Term Memory Note

This closes the visibility gap after Phase 116: ignore-memory receipt compliance is now not only audited by the backend but visible to the group main Agent inside Memory Center. The long-term goal remains active because the larger Claude Code parity target still includes richer typed-memory distillation, automatic rework routing, and more compact lifecycle feedback loops.
