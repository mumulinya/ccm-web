# Phase 280: Model Extraction Semantic Topic Lifecycle

## Status

Completed on 2026-07-14.

Phase 278 admitted evidence-bound model extraction facts into two fixed typed-memory files. Phase 280 replaces those time-independent buckets with deterministic semantic topics that can be reused, split, merged, retired and removed while preserving exact `groupId--gcs_*` session isolation.

## Claude Code Reference

Compared against:

- `D:/claude-code/src/services/extractMemories/prompts.ts`

The reference behavior organizes memory by semantic topic, updates existing topics instead of creating chronological duplicates, removes incorrect or obsolete topics, keeps `MEMORY.md` as a compact index and bounds topic discovery.

CCM follows those lifecycle goals with a stricter trust boundary: the model proposes evidence-bound facts only. Deterministic code owns topic identity, filenames, paths, merging, splitting and deletion.

## Invariants

- Scope remains exactly `groupId--gcs_*`.
- A topic and its facts are never visible to another group session.
- The global Agent receives global context only and does not receive group topic bodies.
- Topic paths are generated by CCM and cannot be supplied by model output.
- Only active evidence-bound facts are rendered into topic files and injected.
- Superseded facts remain in the checksummed archive but disappear from active topic files.
- Legacy `default` sessions are not migrated; deleting them removes their associated artifacts.
- Legacy fixed model extraction files are removed on the next successful typed-memory commit.

## Semantic Topic Lifecycle

Each active `user` or `feedback` fact is assigned to a stable topic using bounded concepts extracted from the admitted fact text. Existing topic identity is reused when the fact already carries a topic or reaches the deterministic similarity threshold.

The lifecycle archive records:

- stable topic ID, display name and safe slug;
- category and bounded semantic concepts;
- active fact checksums and generated document slugs;
- active, retired or merged status;
- merge target, creation, update and retirement timestamps.

Similar historical topics are merged into the older canonical topic. Topics with no remaining active facts become retired. Their generated Markdown files are deleted in the same shared typed-memory artifact transaction as the archive and index update.

## Bounded Storage

- Maximum topics per category: 40, including the consolidated overflow topic.
- Maximum facts per generated topic file: 15.
- Larger topics are split into deterministic `part-N` files.
- Facts beyond the semantic topic capacity enter one deterministic consolidated topic.
- Historical archives already above the cap are deterministically rebalanced on the next commit.
- The archive retains at most 200 lifecycle topic records.
- `MEMORY.md` remains the stable one-line topic index and its loaded projection remains bounded to 200 lines and 25,000 bytes.

The consolidated fact count is recomputed from the current active lifecycle state, so it remains accurate after restart and later commits.

## Atomicity And Cleanup

Topic Markdown updates, retired file deletion, `MEMORY.md` regeneration and the checksummed typed-memory archive commit use the existing shared mutation coordinator and artifact transaction. A crash cannot publish a new archive while leaving an incompatible set of topic files, and startup/read barriers retain the Phase 273 rollback or roll-forward recovery behavior.

The old fixed files are cleaned when encountered:

- `model-extracted-user-constraints.md`
- `model-extracted-corrections.md`

## Memory Center

The Session Memory fleet now reports per session and overall:

- active semantic topics;
- retired semantic topics;
- merged semantic topics;
- facts currently held in consolidated topics.

These counters are read from the checksummed model extraction typed-memory archive and do not expose fact bodies across scopes.
