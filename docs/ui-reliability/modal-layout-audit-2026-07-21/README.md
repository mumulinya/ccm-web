# 全页面弹窗布局审计

## 审计范围

本次扫描了前端组件中的 `modal-overlay` 弹窗，并额外检查了知识库、设置、音乐和宠物空间使用自定义 overlay 的弹窗。通用 overlay 组件共 `21` 个，涉及项目、群聊、任务、代码变更、模板、宠物和工作台。

## 发现的问题

旧弹窗存在两类共同风险：

1. 子组件只定义内容样式，表单宽度、标签布局和操作区依赖父页面 scoped CSS。父级规则无法稳定作用到子组件内部，可能退化成默认浏览器布局。
2. 多个弹窗使用 `420px`、`500px` 或 `650px` 固定最小宽度，但没有自己的移动端覆盖，窄屏会被撑出视口。

高风险组件包括：

- 群聊创建与重命名。
- 模板变量填写。
- 项目 Agent 切换、飞书二维码、目录浏览和共享文件编辑。
- 业务开发任务和需求池。
- 工作台拆分、审核弹窗。
- 宠物与宠物皮肤创建。
- 音乐 Agent 设置。

## 修复

- 在全局样式末尾增加低优先级通用弹窗基线，为旧 `.modal` 提供稳定宽度、最大高度、表单控件和操作区布局。
- `768px` 以下强制清除旧组件的固定 `min-width`，弹窗改为单列底部面板且不超过视口。
- 基线使用 `:where(...)` 降低选择器优先级，已有完整设计的复杂弹窗仍由自身样式控制。
- 项目飞书二维码在窄屏改为上下布局，二维码保持正方形并限制图片尺寸。
- 项目目录浏览器在窄屏缩短目录列表，操作区改为两列按钮。
- 项目共享文件头部在窄屏改为纵向排列。
- 宠物与皮肤创建弹窗增加视口宽高约束。
- 音乐 Agent 设置增加主体滚动和移动端底部面板。

## 无需改造

以下弹窗已有独立、完整的尺寸和响应式规则，本次只验证、不重写：

- 知识库设置与标签编辑。
- 设置中心飞书二维码。
- 群聊测试目标。
- 代码提交面板。
- 项目工具、群聊工具和差异查看器。

## 验证

- Frontend production build 通过。
- 实际“新建群聊”弹窗桌面端宽度 `560px`，输入框和操作区同宽。
- `390 x 844` 下实际群聊弹窗 `min-width: 0`，稳定停靠底部且页面无横向溢出。
- 项目管理桌面与移动端 8 项回归通过。
- 群聊协作既有桌面与移动端回归通过。
- `modal-layout-reliability-selftest.mjs` 固化通用基线和复杂弹窗响应式契约。
- 音乐和宠物旧视觉脚本仍有与本次无关的过期断言：重复歌曲卡片导致 strict locator 冲突、宠物数量不再固定为 2；本次不修改业务来绕过这些断言。
